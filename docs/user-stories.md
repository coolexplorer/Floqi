# Floqi — User Stories

> 기술설계서(`technical-design-document.md`) 기반으로 도출한 유저 스토리 리스트
> 형식: `As a [역할], I want to [행동], so that [이유]`
> 우선순위: P0 (MVP 필수) / P1 (MVP 권장) / P2 (Post-MVP)

## Epic 1: 인증 & 온보딩 (Auth & Onboarding)

| ID | Priority | User Story | Acceptance Criteria |
|----|----------|-----------|-------------------|
| US-101 | P0 | 사용자로서, 이메일/비밀번호로 회원가입할 수 있다. 그래야 Floqi 서비스를 이용할 수 있다. | - Supabase Auth 기반 가입 - 이메일 인증 발송 - 가입 시 profiles 테이블 자동 생성 |
| US-102 | P0 | 사용자로서, 이메일/비밀번호로 로그인할 수 있다. 그래야 내 대시보드에 접근할 수 있다. | - 로그인 성공 시 대시보드로 리다이렉트 - 실패 시 에러 메시지 표시 |
| US-103 | P0 | 사용자로서, Google OAuth로 간편 가입/로그인할 수 있다. 그래야 빠르게 시작할 수 있다. | - Google OAuth 플로우 동작 - 가입과 로그인 모두 지원 |
| US-104 | P1 | 사용자로서, 첫 로그인 시 온보딩 과정을 거칠 수 있다. 그래야 초기 설정(타임존, 언어)을 할 수 있다. | - 타임존 선택 - 선호 언어 설정 - onboarding_completed 플래그 업데이트 |
| US-105 | P0 | 사용자로서, 로그아웃할 수 있다. 그래야 내 계정을 안전하게 관리할 수 있다. | - 세션 종료 - 랜딩 페이지로 리다이렉트 |

## Epic 2: 서비스 연결 (Service Connections)

| ID | Priority | User Story | Acceptance Criteria |
|----|----------|-----------|-------------------|
| US-201 | P0 | 사용자로서, Google 계정을 OAuth로 연결할 수 있다. 그래야 Gmail과 Calendar 자동화를 사용할 수 있다. | - OAuth consent 화면 → 콜백 처리 - access_token, refresh_token 암호화 저장(AES-256-GCM) - scopes: gmail.readonly, gmail.send, calendar.readonly |
| US-202 | P1 | 사용자로서, Notion 계정을 OAuth로 연결할 수 있다. 그래야 Notion 기반 자동화를 사용할 수 있다. | - Notion OAuth 플로우 - 토큰 암호화 저장 |
| US-203 | P0 | 사용자로서, 연결된 서비스 목록을 확인할 수 있다. 그래야 어떤 앱이 연결되어 있는지 알 수 있다. | - Connections 페이지에서 연결 상태 표시 - 활성/비활성 상태 구분 |
| US-204 | P0 | 사용자로서, 연결된 서비스를 해제할 수 있다. 그래야 더 이상 사용하지 않는 연결을 제거할 수 있다. | - 연결 해제 시 토큰 삭제 - 관련 자동화 비활성화 경고 |
| US-205 | P1 | 사용자로서, 만료된 OAuth 토큰이 자동으로 갱신된다. 그래야 자동화가 중단 없이 실행된다. | - refresh_token으로 자동 갱신 - 갱신 실패 시 사용자에게 재연결 요청 알림 |

## Epic 3: 자동화 관리 (Automation CRUD)

| ID | Priority | User Story | Acceptance Criteria |
|----|----------|-----------|-------------------|
| US-301 | P0 | 사용자로서, 사전 제공된 템플릿 목록에서 자동화를 선택하여 생성할 수 있다. 그래야 쉽게 시작할 수 있다. | - 5개 MVP 템플릿 표시 (Morning Briefing, Email Triage, Reading Digest, Weekly Review, Smart Save) - 템플릿 선택 → 기본 설정으로 자동화 생성 |
| US-302 | P1 | 사용자로서, 자연어로 원하는 자동화를 설명하여 커스텀 자동화를 만들 수 있다. 그래야 나만의 워크플로우를 구성할 수 있다. | - 자연어 입력 필드 - agent_prompt로 저장 - 필요한 서비스 연결 안내 |
| US-303 | P0 | 사용자로서, 내 자동화 목록을 확인할 수 있다. 그래야 현재 활성화된 자동화를 관리할 수 있다. | - 자동화 카드 리스트 표시 - 이름, 아이콘, 상태, 마지막 실행 시간, 다음 실행 시간 |
| US-304 | P0 | 사용자로서, 자동화의 스케줄(크론 표현식)을 설정할 수 있다. 그래야 원하는 시간에 실행되도록 할 수 있다. | - trigger_type: cron 설정 - 사용자 친화적 UI (매일 오전 7시, 매주 월요일 등) → 크론 표현식 변환 |
| US-305 | P0 | 사용자로서, 자동화를 활성화/일시정지할 수 있다. 그래야 필요에 따라 실행을 제어할 수 있다. | - status: active ↔ paused 토글 - 일시정지 시 스케줄러에서 제외 |
| US-306 | P0 | 사용자로서, 자동화를 삭제할 수 있다. 그래야 불필요한 자동화를 정리할 수 있다. | - 삭제 확인 모달 - 관련 실행 로그는 유지 |
| US-307 | P1 | 사용자로서, 자동화의 프롬프트와 설정을 수정할 수 있다. 그래야 동작을 세부 조정할 수 있다. | - agent_prompt 편집 - trigger_config 수정 - agent_config 수정 |
| US-308 | P1 | 사용자로서, 자동화를 즉시 수동 실행할 수 있다. 그래야 결과를 바로 확인하고 테스트할 수 있다. | - "Run Now" 버튼 - trigger_type: on_demand로 즉시 실행 - 실행 결과 확인 |

## Epic 4: 자동화 템플릿 (Templates)

| ID | Priority | User Story | Acceptance Criteria |
|----|----------|-----------|-------------------|
| US-401 | P0 | 사용자로서, **Morning Briefing** 템플릿을 사용할 수 있다. 그래야 매일 아침 오늘 일정 + 중요 이메일 + 날씨 요약을 받을 수 있다. | - Google Calendar 오늘 일정 조회 - Gmail 중요 이메일 요약 - 날씨 정보 포함 - 이메일로 결과 전송 |
| US-402 | P0 | 사용자로서, **Email Triage** 템플릿을 사용할 수 있다. 그래야 읽지 않은 이메일을 AI가 분류/요약해 줄 수 있다. | - 미읽은 이메일 조회 - 긴급/중요/참고 분류 - 요약 리스트 생성 |
| US-403 | P0 | 사용자로서, **Reading Digest** 템플릿을 사용할 수 있다. 그래야 관심 분야의 뉴스를 요약 받을 수 있다. | - 사용자 관심 카테고리 기반 뉴스 수집 - 주요 기사 요약 - Notion 페이지 또는 이메일로 전달 |
| US-404 | P1 | 사용자로서, **Weekly Review** 템플릿을 사용할 수 있다. 그래야 한 주간의 활동을 정리 받을 수 있다. | - 주간 실행 로그 집계 - 캘린더 일정 회고 - 이메일 통계 |
| US-405 | P1 | 사용자로서, **Smart Save** 템플릿을 사용할 수 있다. 그래야 특정 이메일/뉴스를 Notion에 자동 저장할 수 있다. | - Webhook 트리거로 동작 - 이메일/뉴스 내용 → Notion 페이지 생성 |

## Epic 5: 자동화 실행 엔진 (Execution Engine)

| ID | Priority | User Story | Acceptance Criteria |
|----|----------|-----------|-------------------|
| US-501 | P0 | 시스템으로서, Cron 스케줄에 따라 자동화를 자동 실행한다. 그래야 사용자가 설정한 시간에 자동화가 동작한다. | - CronDispatcher가 매분 due automations 체크 - Asynq 큐에 태스크 enqueue - next_run_at 업데이트 |
| US-502 | P0 | 시스템으로서, AI Agent가 Tool Use 루프를 통해 MCP 도구를 호출한다. 그래야 이메일, 캘린더, 뉴스 등 외부 데이터를 수집하고 작업을 수행한다. | - Anthropic Messages API 호출 - tool_use → tool_result 반복 (최대 10회) - end_turn 시 결과 반환 |
| US-503 | P0 | 시스템으로서, 자동화 실행 결과를 execution_logs에 기록한다. 그래야 실행 이력을 추적할 수 있다. | - 실행 시작/완료 시간 기록 - tool_calls, tokens_used, output 저장 - 성공/실패 상태 기록 |
| US-504 | P0 | 시스템으로서, 실행 실패 시 최대 3회 재시도한다. 그래야 일시적 오류에 대응할 수 있다. | - Asynq MaxRetry(3) 설정 - 재시도 간 지수 백오프 - 최종 실패 시 error 상태 저장 |
| US-505 | P1 | 시스템으로서, 사용자의 Managed/BYOK LLM 설정에 따라 적절한 API 키를 사용한다. 그래야 각 사용자에게 맞는 LLM 비용 모델을 적용할 수 있다. | - profile.llm_config.mode 확인 - managed: 서비스 API 키 사용 - byok: 사용자 암호화된 키 복호화 후 사용 |
| US-506 | P1 | 시스템으로서, Webhook 이벤트를 수신하여 자동화를 트리거한다. 그래야 실시간 이벤트 기반 자동화가 가능하다. | - Next.js API Route로 webhook 수신 - Redis를 통해 Go Worker에 전달 - trigger_type: webhook 처리 |

## Epic 6: 실행 로그 & 모니터링 (Logs & Monitoring)

| ID | Priority | User Story | Acceptance Criteria |
|----|----------|-----------|-------------------|

| US-604 | P2 | 사용자로서, 대시보드에서 자동화 실행 통계를 한눈에 볼 수 있다. 그래야 전체 사용 현황을 파악할 수 있다. | - 활성 자동화 수 - 이번 주 실행 횟수 - 총 사용 토큰 - 성공률 |

| US-1006 | P1 | 시스템으로서, 실행 로그에서 민감 정보를 마스킹한다. 그래야 로그 노출 시에도 개인정보가 보호된다. | - 이메일 주소, 토큰 마스킹 - 비밀번호, API 키 제외 |
